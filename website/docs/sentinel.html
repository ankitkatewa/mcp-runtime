Sentinel

mcp-sentinel policy, audit, and observability on the MCP request path.

The services/ directory contains the bundled service stack for proxy enforcement, audit, query, governance UI, and observability around MCP servers. It is where live MCP requests are governed and observed around the runtime, not arbitrary cluster traffic, not just a dashboard, and not just a single proxy.

Runtime docs Analytics API

Services

mcp-proxy

Transparent sidecar that extracts identity, evaluates tool-level policy, emits allow or deny audit events, and forwards traffic upstream to the MCP server.

ingest

Receives POST /events, validates API keys or optional JWTs, and writes events to Kafka.

processor

Consumes Kafka, batches events, and writes them into ClickHouse with indexed audit fields for filtering.

api

Exposes analytics endpoints, dashboard summaries, runtime governance APIs for creating or updating grants/sessions, and component operations.

ui

Three-tab dashboard: overview metrics and events, governance forms for creating grants/sessions plus enable/disable/revoke actions, and operations for component health and safe restarts.

gateway

Kubernetes gateway deployment that fronts the sentinel API, ingest, and UI surfaces.

reference mcp-server

The repo also includes a small example MCP server so sidecar, ingest, and smoke-test flows can be exercised end to end.

Event path

01. Proxy evaluates the request

The sidecar reads identity headers, loads policy data rendered from MCPAccessGrant and MCPAgentSession CRs, checks trust, and decides allow or deny at tool-call time.

02. Ingest receives the event

The event is sent to /events on the ingest service and written into the Kafka topic mcp.events.

03. Processor batches to ClickHouse

The processor reads Kafka, batches messages, and writes them to the ClickHouse-backed event table.

04. API exposes query surfaces

The API returns recent events, stats, sources, event types, and filtered audit views over the stored event stream.

05. UI and dashboards consume the data

The UI shows the stream directly, while Grafana, Prometheus, Tempo, Loki, and Promtail cover the broader observability path.

Storage and observability

ClickHouse

Stores the event stream with materialized fields for server, namespace, cluster, human, agent, session, decision, and tool name.

Kafka and Zookeeper

Back the ingestion pipeline so events can be buffered before the processor writes them to ClickHouse.

Prometheus and Grafana

The manifests include scrape configuration and a Grafana deployment so service metrics are part of the out-of-box story.

OTel, Tempo, Loki, Promtail

Trace and log plumbing are part of the same install path so the stack goes beyond one audit table.

Governance UI walkthrough

The UI's Governance tab creates and operates the same MCPAccessGrant and MCPAgentSession resources that the CLI manages. You can also reach these flows directly through the Runtime Governance API documented under API → Runtime Governance.

Create a grant

Click Create Grant on the Governance tab. Required fields: name, namespace, server, and at least one of human ID or agent ID. Tool rules use one rule per line in the form tool:allow or tool:allow:trust.

list_invoices:allow
refund_invoice:allow:high
Create a session

Click Create Session. Pick a consented trust level and an optional expiry. The session is what the gateway looks up at tools/call time alongside the grant.

Disable or enable a grant

Each row exposes a single action. Disable flips spec.disabled=true on the CR — the grant remains for audit, but the gateway treats it as denying. Re-enable to flip it back.

Revoke or unrevoke a session

The same single-action row pattern toggles spec.revoked. A revoked session denies subsequent tools/call requests immediately at the gateway.

Filter the lists

The filter input on each table searches server name, human ID, and agent ID. Filtering is local to the loaded set, so refresh first if you suspect the cluster has changed.

CLI parity

The same CRUD flows are available as mcp-runtime access grant and mcp-runtime access session. The UI is a convenience layer; CRs are the source of truth.

Runtime integration

mcp-runtime setup builds the sentinel images and deploys this stack by default. Use --without-sentinel to skip it.

Repository structure

Flattened structure

Services are now in services/, manifests in k8s/, and shared libraries in pkg/. This replaces the nested mcp-sentinel/ directory.

Shared libraries

New pkg/access, pkg/sentinel, pkg/clickhouse, and pkg/k8sclient packages are used by both CLI and API services.

Compatibility aliases remain

The runtime and sample MCP wiring still accept older MCP_ANALYTICS_* environment names so existing ingest configuration does not break during the rename.

Companion plane, not an addon afterthought

First-class part of the product

Sentinel is where analytics, gateway behavior, and lightweight policy-enforcement services live around MCP servers. That is why the website and docs now treat it as a first-class part of the product instead of a side note.
